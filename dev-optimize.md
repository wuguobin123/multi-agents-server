# 多 Agent 项目优化与迭代方案

## 1. 文档目的

本方案用于将当前仓库从“可运行的多 Agent MVP”逐步完善为“结构清晰、依赖标准、可扩展、可部署、可验证”的标准多 Agent 架构项目。

本方案先给出迭代方向和实施顺序，不直接进入详细开发任务拆分。

## 2. 当前项目现状

结合仓库当前实现，项目已经具备最小可运行能力，但整体仍处于 MVP 骨架阶段。

### 2.1 已具备能力

1. 已有 FastAPI 对外接口。
2. 已有 LangGraph 工作流，可完成 `planner -> route -> agent -> reflect -> finish` 的基本编排。
3. 已有四类基础 Agent：
   - `planner`
   - `qa_agent`
   - `tool_agent`
   - `fallback_agent`
4. 已有统一 Tool Registry，支持本地 Skill Tool 和 HTTP 形式的 MCP Tool 适配。
5. 已有基础 RAG 能力，可以从本地知识库目录构建切块清单并完成简单检索。
6. 已有基础单元测试和集成测试。
7. 已提供 Docker 镜像和最小 `docker-compose.yml`，可以启动应用容器。

### 2.2 当前主要不足

1. 项目分层仍偏薄，业务编排、运行时状态、持久化边界不够清晰。
2. Planner 仍是关键词规则判断，不是标准的结构化规划能力。
3. Tool 体系只有最小注册和调用能力，缺少参数治理、权限治理、重试、熔断和审计。
4. RAG 仍基于本地文件和 `index.json` 清单，不是标准向量检索架构。
5. 没有结构化数据存储，运行轨迹、会话数据、任务状态无法持久化。
6. Docker 编排只有应用容器，没有数据库、向量库、健康检查和初始化流程。
7. 可观测性较弱，缺少面向运行排障的指标、trace 和失败分类。
8. 测试还没有覆盖真实容器依赖、数据初始化、向量检索链路和迁移脚本。

## 3. 优化目标

本次优化目标不是简单“补两个中间件”，而是建立一套完整的标准化多 Agent 项目形态。

目标形态如下：

1. API 层、运行时层、Agent 层、Tool 层、RAG 层、持久化层、观测层职责清晰。
2. MySQL 用于结构化数据持久化。
3. 向量数据库用于 RAG 的 Embedding 和检索存储。
4. Agent 编排可扩展，不依赖硬编码关键词规则作为长期方案。
5. Docker 开发环境可一键拉起应用、MySQL、向量库及必要依赖。
6. 本地开发、测试验证、部署运行三条链路一致。
7. 具备最小可用的工程标准件：
   - 配置管理
   - 数据迁移
   - 健康检查
   - 初始化脚本
   - 冒烟验证
   - 可观测日志

## 4. 目标架构

### 4.1 分层设计

建议将项目收敛为以下结构：

1. `app/api`
   - 接收 HTTP 请求。
   - 鉴权、请求校验、响应序列化。
   - 输出统一错误码与 trace。

2. `app/runtime`
   - 作为应用编排入口。
   - 管理会话上下文、超时、状态机执行、结果聚合。
   - 负责组装 Planner、Agent、Tool、RAG、Repository。

3. `app/graph`
   - 仅负责编排节点和边。
   - 不直接持有具体基础设施细节。

4. `app/agents`
   - 每个 Agent 仅关注自身职责。
   - 通过统一接口接入模型、检索、工具和持久化服务。

5. `app/tools`
   - 统一 Tool 协议。
   - 支持 Skill、本地函数、HTTP MCP、后续外部服务。

6. `app/rag`
   - 文档导入。
   - 文本切分。
   - Embedding。
   - 向量入库。
   - 检索与引用整理。

7. `app/repositories`
   - 封装 MySQL 访问。
   - 管理会话、消息、任务记录、工具调用记录、知识库元数据。

8. `app/observability`
   - 统一日志。
   - Request ID / Session ID 贯穿。
   - Agent trace、tool trace、错误分类、耗时统计。

### 4.2 数据存储职责

建议明确区分两类存储：

1. MySQL
   - 存储结构化数据。
   - 例如：
     - 会话信息
     - 历史消息
     - Planner 结果摘要
     - Agent 执行轨迹
     - Tool 调用记录
     - 知识库文档元数据
     - 导入任务状态

2. 向量数据库
   - 存储文档切块后的向量。
   - 用于 RAG 检索。
   - 不负责业务事务型结构化数据。

## 5. 选型建议

### 5.1 结构化数据库

建议保留 MySQL 作为结构化数据存储，原因如下：

1. 适合会话、消息、任务状态等事务型数据。
2. 生态成熟，便于迁移、备份和排障。
3. 适合与 Alembic 或类似迁移机制配合。

### 5.2 向量数据库

建议优先选择独立向量数据库，不建议继续沿用本地 `index.json` 方案。

建议优先级如下：

1. 第一优先：Qdrant
   - Docker 友好。
   - 接口清晰。
   - 适合中小型 RAG 项目。
   - 与 MySQL 组合清晰，不会混淆职责。

2. 第二优先：Chroma
   - 上手简单。
   - 适合快速验证。
   - 但后续生产级治理和独立运维能力弱于 Qdrant。

3. 暂不建议：继续使用本地文件伪索引
   - 不利于扩展。
   - 无法形成标准 RAG 存储链路。

说明：

如果坚持使用 MySQL 存结构化数据，则不建议同时再引入 `pgvector` 路线，否则整个项目会变成两套数据库范式并存，复杂度更高。

## 6. 分阶段迭代方案

### 阶段一：项目结构标准化

目标：

先把当前 MVP 代码整理为可扩展的工程结构，为后续接入 MySQL 和向量库打基础。

主要工作：

1. 明确 API、Runtime、Graph、Agent、Tool、RAG、Repository 的边界。
2. 从当前 `runtime.py` 中拆出更清晰的依赖装配逻辑。
3. 统一 Schema：
   - 会话请求
   - Agent 执行结果
   - Tool 调用结果
   - Trace 输出
4. 为后续持久化准备领域对象：
   - `Session`
   - `Message`
   - `AgentRun`
   - `ToolCall`
   - `KnowledgeDocument`
   - `KnowledgeChunk`
5. 统一错误分类和返回格式。

阶段产出：

1. 更清晰的目录结构。
2. 可扩展的依赖注入和组件装配。
3. 后续接入数据库不需要反复改 graph 和 agent 接口。

### 阶段二：接入 MySQL 持久化

目标：

让系统具备最小结构化持久化能力，解决运行状态全在内存中的问题。

主要工作：

1. 引入 ORM 或数据库访问层。
2. 新增数据库配置项和环境变量。
3. 新增迁移机制。
4. 建立核心表结构：
   - `chat_session`
   - `chat_message`
   - `agent_run`
   - `tool_call`
   - `knowledge_document`
   - `knowledge_chunk`
   - `ingestion_job`
5. 在请求链路中持久化：
   - 会话创建
   - 用户输入
   - Planner 输出
   - Agent 执行轨迹
   - Tool 调用结果
6. 支持后续按 `session_id` 回溯执行过程。

阶段产出：

1. MySQL 容器和连接配置。
2. 迁移脚本。
3. 基础 Repository 层。
4. 请求闭环中的结构化数据落库。

### 阶段三：RAG 向量化升级

目标：

将当前基于本地切块文件的检索逻辑，升级为标准化 RAG 链路。

主要工作：

1. 拆分 RAG 组件：
   - Loader
   - Splitter
   - Embedder
   - VectorStore
   - Retriever
2. 将文档导入任务改造成可重复执行的 ingest 流程。
3. 文档元数据存 MySQL，向量存 Qdrant。
4. 补充 Embedding 配置与抽象层。
5. 检索结果支持：
   - Top-K
   - metadata 过滤
   - 引用回传
   - 检索为空时显式降级
6. 保留本地 mock 模式，避免开发环境完全依赖真实外部模型。

阶段产出：

1. 标准向量存储链路。
2. 可重复导入的知识库流程。
3. RAG 检索能力不再依赖本地 `index.json`。

### 阶段四：Agent 编排能力升级

目标：

让多 Agent 从“规则路由”升级为“结构化规划 + 可追踪执行”。

主要工作：

1. 将 Planner 从关键词规则升级为模型驱动的结构化规划输出。
2. 增加 Planner 输出 Schema 校验。
3. 增加 Agent 能力注册表，而不是在 graph 中写死名称。
4. 增加 Tool 选择策略：
   - 基于描述匹配
   - 基于 planner hints
   - 基于参数 schema
5. 增加失败处理：
   - 超时
   - 重试
   - 熔断
   - fallback 退化
6. 预留后续可扩展 Agent：
   - `critic_agent`
   - `retrieval_agent`
   - `summarizer_agent`

阶段产出：

1. Planner 更稳定。
2. Agent 扩展更方便。
3. 工具链路不再只适合演示。

### 阶段五：Tool 体系工程化

目标：

把当前最小 Tool Registry 升级成可维护的工具框架。

主要工作：

1. 增加 Tool 输入校验。
2. 增加 Tool 调用权限和开关管理。
3. 增加调用超时、重试、熔断策略。
4. 增加统一日志和审计字段。
5. 为外部 MCP Tool 建立更稳定的协议适配。
6. 支持根据配置动态启停工具。

阶段产出：

1. Tool 使用更安全。
2. Tool 失败更可排查。
3. 后续接外部系统时不需要重写一套调用逻辑。

### 阶段六：Docker 开发环境与部署标准化

目标：

形成标准的本地开发和部署环境，不再只有单应用容器。

建议 Docker 组成如下：

1. `app`
2. `mysql`
3. `qdrant`

可选：

4. `adminer` 或 `phpmyadmin`，仅开发环境使用
5. `otel-collector` 或日志采集组件，按后续需要决定

主要工作：

1. 扩展 `docker-compose.yml`。
2. 为 MySQL、Qdrant 配置持久化卷。
3. 增加 `depends_on` 和健康检查。
4. 增加初始化脚本：
   - 数据库建表迁移
   - 知识库导入
5. 完善 `.env.example`。
6. 提供一键启动、重建、回收命令。

阶段产出：

1. 本地开发环境完整。
2. 新成员可直接启动项目。
3. 部署验证不再依赖手工补环境。

### 阶段七：验证、观测与发布闭环

目标：

让项目具备可回归、可排障、可发布能力。

主要工作：

1. 增加数据库迁移测试。
2. 增加知识库导入测试。
3. 增加容器级集成测试。
4. 增加 `/healthz` 之外的 readiness 检查。
5. 增加结构化日志字段：
   - request_id
   - session_id
   - plan_intent
   - selected_agents
   - tool_name
   - latency_ms
   - error_type
6. 增加最小冒烟脚本：
   - 健康检查
   - QA 请求
   - Tool 请求
   - RAG 导入检查

阶段产出：

1. 本地、测试、部署都可重复验证。
2. 故障定位路径清晰。
3. 项目具备持续迭代基础。

## 7. 推荐实施顺序

建议按以下顺序推进：

1. 阶段一：项目结构标准化
2. 阶段二：MySQL 持久化
3. 阶段三：RAG 向量化升级
4. 阶段四：Agent 编排能力升级
5. 阶段五：Tool 体系工程化
6. 阶段六：Docker 开发环境与部署标准化
7. 阶段七：验证、观测与发布闭环

不建议一开始同时改以下内容：

1. 数据库
2. 向量库
3. Planner
4. Tool 协议
5. Docker 编排

原因：

这几块同时动会导致排查成本过高，问题来源不清晰，回归验证也会失控。

## 8. 近期优先级建议

如果按“最小风险、最高收益”排序，建议近期先做：

1. 重构项目分层。
2. 接入 MySQL 持久化。
3. 补 Docker 中的 MySQL 服务。
4. 定稿向量库选型，优先 Qdrant。
5. 再升级 RAG 链路。

也就是说，MySQL 和向量库都应该纳入方案，但不是第一步同时硬接。

## 9. 风险与取舍

### 9.1 主要风险

1. 若过早引入多个基础设施，开发复杂度会明显上升。
2. 若先改数据库而不整理代码边界，后续会造成 Runtime、Agent、Repository 相互耦合。
3. 若继续保留当前本地伪索引方案过久，会拖慢后续 RAG 标准化。
4. 若没有迁移机制和容器初始化流程，Docker 环境很快会变得不可维护。

### 9.2 关键取舍

1. 结构化数据和向量数据必须分层设计，不要混为一谈。
2. 开发阶段可以保留 `mock provider`，降低联调门槛。
3. 对外部模型和外部工具的接入应通过适配层完成，避免侵入 Agent 主逻辑。

## 10. 验收标准

当以下条件满足时，可认为项目完成了从 MVP 到标准化多 Agent 项目的第一轮升级：

1. 本地 Docker 可拉起 `app + mysql + qdrant`。
2. 应用启动后可自动完成数据库迁移或明确给出初始化步骤。
3. `/healthz` 和 readiness 检查均可正常返回。
4. `/v1/chat` 的 QA 链路可以基于向量库返回答案和引用。
5. `/v1/chat` 的 Tool 链路可以完成至少一次真实工具调用。
6. 会话、消息、执行轨迹、工具调用可以在 MySQL 中查询。
7. 知识库文档可完成导入、重建和重查。
8. 基础自动化测试和容器级冒烟测试通过。

## 11. 下一步建议

本方案之后，建议继续补一份“实施任务拆分文档”，按阶段拆成明确开发项，例如：

1. 数据模型设计
2. Docker 改造任务
3. Repository 接入任务
4. RAG 重构任务
5. 测试补齐任务

这样可以直接进入逐项开发，而不是停留在方向讨论层。
